# stock-app

## プロジェクト概要

株式・投資情報を管理・表示するWebアプリケーション。

## 技術スタック

### フロントエンド
- **React** 18.2.0 — UIフレームワーク
- **Vite** 5.0.8 — ビルドツール・開発サーバー
- **Recharts** 2.10.0 — チャート描画ライブラリ

### バックエンド（データ取得）
- **Google Apps Script（GAS）** — Yahoo Finance からの株価データ取得（日本株・米国株対応）
- **Alpha Vantage API** — 米国株データ取得（フォールバック用、無料プラン25回/日）

### データソース
- **Yahoo Finance** `query1.finance.yahoo.com/v8/finance/chart/` — 株価履歴データ
- **Yahoo Finance Japan** `query.finance.yahoo.co.jp/v8/finance/chart/` — 日本株の会社名取得

### インフラ・デプロイ
- **GitHub Pages** — 静的サイトホスティング（公開URL: https://nori1975461.github.io/stock-app）
- **gh-pages** 6.1.1 — デプロイツール（`npm run deploy` で公開）

### 予測ロジック概要（CLEAR TRADE理論）
- **MA/RSI/MACD/ボリンジャー等の従来指標は一切使用しない**
- 出来高＋純粋チャート分析の9指標でスコアリング
- stableScore（7指標・週次安定）でランキング、score（9指標）で詳細表示
- CT理論の4次元構造をすべて実装済み（次元1〜4）

## 主要機能

- **次元1：相場環境判断** — 米国3＋日本13セクター代表銘柄をスキャンし EASY/NORMAL/HARD を判定
- **次元2：先導株分析** — CT_LEADERS 8銘柄のリアルタイムCT分析（モナリザ理論）
- **次元3：CT銘柄スクリーニング** — CT_UNIVERSE 60銘柄を一括スキャンして上位10銘柄を抽出
- **次元4：エントリータイミング** — 詳細ビューに初速判定・2日目確認を表示
- 単一銘柄の詳細分析（CT9指標・分散エグジット判断・1か月後見通し・チャート）
- 複数銘柄（最大**10**銘柄）の上昇予測ランキング＋購入推奨カード
- 分析期間：1週間〜12週間（6段階）
- GAS URL・Alpha Vantage APIキーを localStorage に保存（再入力不要）
- 日本株（.T）・米国株の両対応

## 既知の制限・注意事項

- 日本語会社名はGASの静的マッピング（約130銘柄）で対応済み。未収録銘柄は英語名表示
- Alpha Vantage 無料プランは25回/日の上限あり（複数銘柄分析で消費が早い）
- Yahoo Finance は非公式APIのため、予告なく仕様変更・遮断の可能性あり
- 予測ロジックはCT理論（出来高＋チャート）のみ（ファンダメンタルズ・ニュース等は未考慮）
- 投資判断の根拠としての利用は不可（参考情報のみ）

## CT_UNIVERSE（スクリーニング対象）の定期見直しルール

### 見直しのタイミング

**四半期に1回**（3か月ごと）を目安に以下を確認する。

### 追加の基準（以下の条件を原則すべて満たすこと）

1. **東証プライム上場** — グロース・スタンダードは原則対象外
2. **時価総額の規模** — 機関投資家が参加できる規模（目安：時価総額1,000億円以上）
3. **セクター連動性** — 同セクターの他銘柄と連動して動く構造を持つこと
4. **OBVの確立** — 上場から2年以上経過し、機関投資家の参加パターンが価格履歴に現れていること
5. **銘柄数の上限** — 100〜120銘柄を上限とする（GASのスキャン時間と品質のバランス）

### 削除の基準

- 上場廃止・合併・業種転換など構造変化があった銘柄
- 複数四半期にわたってスコアが機能しない（出来高不足・価格停滞）銘柄

### 「高騰株をCT_UNIVERSEに追加すべきか」の判断フロー

株価が急騰している銘柄を見たとき、以下の順序で判断する：

```
① その銘柄は東証プライムか？
   → NO（グロース・スタンダード）→ 追加しない。CT理論の規律可能性パターンが出にくい

② 急騰の理由は何か？
   → 個別材料（決算・契約・新製品）→ 追加しない。一時的スパイクはCT理論のシグナルではない
   → 構造的テーマ（AI・防衛・エネルギー転換など）→ 次へ

③ 時価総額・機関投資家参加・セクター代表性を満たすか？
   → NO → 追加しない
   → YES → 四半期見直しのタイミングで追加を検討する
```

**「高騰しているから追加する」はCT理論と逆。「CT基準を満たす銘柄を先に選んでおく」がCT理論の本質。**

### 過去の教訓（2026-05-20）

QPSホールディングス（464A.T）・QDレーザ（6613.T）が高騰した際に追加を検討したが、
両銘柄とも東証グロース・小型株・個別材料起因のスパイクであり、CT理論の4条件を満たさないと判断。
追加しなかった。→ この判断フローを上記に整備した。

## 先導株（CT_LEADERS）の選定基準と見直しルール

### 選定基準（以下の条件を原則すべて満たすこと）

1. **時価総額の大きさ** — 機関投資家が大量に買える規模であること
2. **セクター代表性** — その銘柄が動くとセクター全体が動く構造にあること
3. **機関投資家の参加** — 長期OBVが上昇傾向にあること（継続的な機関買いの証拠）
4. **価格リーダーシップ** — 相場転換時に他銘柄より先に動く実績があること

### 先導株と一般CT銘柄の違い

- **先導株（CT_LEADERS）** = 市場を「動かす」銘柄。静的候補リスト（月次〜四半期単位で見直す）
- **CT銘柄（CT_UNIVERSE）** = 先導株の動きに「乗る」銘柄。動的スクリーニング対象

この2つを混同しない。CT_LEADERSを60銘柄に拡大することは設計を壊す（次元2と次元3の区別がなくなる）。

### 見直しのタイミング

アプリの品質スコア（★0〜5）が**2点以下（⚠ 要見直し）を複数週連続**で示した銘柄は、以下の観点で人間が判断する：

- その銘柄は今も「市場を動かす力」を持っているか
- より適切な代替銘柄（同セクターでより機関注目度が高い銘柄）はあるか
- 相場構造の変化（例：新しいセクターが台頭）を反映すべきか

**自動入れ替えはしない。最終判断は必ず人間が行う。**

## Git 運用ルール

**コードを変更するたびに必ずGitHubへプッシュすること。**

```bash
git add <変更ファイル>
git commit -m "変更内容を簡潔に記述"
git push origin main
```

### 基本方針

- コミットは意味のある単位で行う（機能単位、バグ修正単位など）
- コミットメッセージは日本語でも英語でも可。変更の「なぜ」を明記する
- `main` ブランチへ直接プッシュする（小規模プロジェクトのため）
- 破壊的な変更（ファイル削除、大規模リファクタ）は事前にユーザーへ確認する
- `--no-verify` や `--force` は使用しない（明示的な指示がある場合を除く）

### コミットの流れ

1. `git status` で変更ファイルを確認
2. 関係するファイルを個別に `git add` でステージング（`git add -A` は避ける）
3. `git commit -m "..."` でコミット
4. `git push origin main` でGitHubへプッシュ

## CLEAR TRADE理論（CT理論）実装原則

**このアプリはCT理論（Mr.K）に基づく。MA/RSI/MACD/ボリンジャー等の従来指標はスコアに使用しない。**

新しいCT指標を実装・改善する際は、必ず以下の3点を確認すること：

1. **この指標は何を測っているか**
   - 「銘柄の質」= 中長期トレンドの強さ・継続性 → スクリーニングスコアに含める
   - 「エントリータイミング」= 今この瞬間に入るかどうかの確認 → 詳細ビューのみ

2. **この指標の時間軸はどこか**
   - 週次評価（複数日〜数週間で形成）→ スクリーニングスコアに含めてよい
   - 日次・一度限り（エントリー前の瞬間判断）→ スクリーニングスコアに含めてはならない

3. **スクリーニングスコアに含めるべきか、詳細ビューに留めるべきか**
   - スクリーニングスコア（`stableScore`）= 銘柄の質指標のみ。ランキング・先導株ソートに使用
   - 詳細ビュー = エントリータイミング指標（2日目確認・初速判定）を「エントリータイミング」セクションとして表示

**この判断を怠ると、エントリータイミング指標が毎日再計算されてランキングが日次で大きく変動する。**
これはCT理論の「銘柄スクリーニングは週次で行う」という原則に反する。

4. **この表示・実装はCT理論の定義と整合しているか（必須の最終確認）**

   新しい表示・機能・データ構造を実装する前に、必ず以下を自問すること：
   - CT理論における「この概念の正確な定義」は何か
   - 実装が「その定義を正確に反映しているか」
   - 表示された結果がCT理論の定量的・定性的な記述と矛盾しないか

   **この確認を怠ることが、概念の混同・フレームドリフト・理論との不整合の最大の原因となる。**
   技術的に動くことと、CT理論として正しいことは別問題である。

CT理論の2段階構造：
```
Stage 1：銘柄スクリーニング（週次）
  出来高3日平均・OBV15日・規律可能性15日・マグネット効果90日
  停滞シグナル・連続日数・ローソク足10日 → stableScore

Stage 2：エントリータイミング確認（日次・一度限り）
  初速判定・2日目確認 → 詳細ビュー専用
```

## 大きな間違いが発覚したときのプロセス（必須）

**実装に入る前に、必ず根本原因の検証を行うこと。**

大きな間違い（概念的矛盾・理論との不整合・設計ミス）が見つかったとき：

1. **なぜ間違えたのかを徹底的に検証する**（「何が間違っていたか」ではなく「なぜ間違えたか」）
2. **ユーザーに根本原因を説明する**（表面的なバグ報告ではなく、思考プロセスのどこが崩れたかを明示）
3. **根本原因を踏まえた上で実装する**（同じ種類の間違いを再発させないための設計を意識する）

この順序を省略して「解決策をすぐ実装する」ことは禁止。すぐ直すことよりも、なぜ間違えたかを理解することの方が価値が高い。

### 過去の教訓：先導株の概念混同（2026-05-16）

「CT_LEADERSの全銘柄 = 今現在の先導株」として表示した誤りの根本原因：

- **静的候補リスト**（先導株素質のある銘柄 = 週次・不変）と  
  **動的アクティブ状態**（今日現在、市場を牽引している銘柄 = 日次・変化）を  
  1つの変数・1つのパネルに混在させた
- 「CT_LEADERS」という名前が「今リードしている」を意味するのに、実態は「候補」だった
- 機能拡張のたびにパネルの意味が変質していたのに気づかなかった（フレームドリフト）
- 実装前に「この概念は何か」を定義せず、曖昧なまま作り始めた

**再発防止の原則：新しいデータ構造や機能を作る前に「これは静的か動的か」「週次か日次か」「候補か状態か」を明確に定義すること。**
